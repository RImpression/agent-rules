# Agent Rules Enhance

## 目标

在 CLI 生成 `.agents/` 骨架后，利用 AI 的代码理解能力，将通用模板升级为**精准的、针对当前项目**的规范内容。

## 触发条件

以下任一条件满足时激活此 Skill：

1. 用户说"初始化规范"、"补充规范"、"增强规范"
2. 用户说"更新 glossary"、"生成 module-map"、"分析项目结构"
3. 检测到 `.agents/` 目录存在但内容为模板占位（如 glossary 中有 `<!-- 按需补充 -->` 注释）
4. 用户在对话中提到了一个业务词，但 glossary.md 中未收录

## 执行步骤

### 阶段一：深度扫描（init 后首次触发）

1. **读取已有骨架**
   - 读取 `.agents/overviews/overview.md` 获取已扫描的基础信息
   - 读取 `.agents/overviews/glossary.md` 确认当前映射情况
   - 读取 `.agents/rules/rule.md` 确认已有规则清单

2. **分析源码结构**
   - 扫描 `src/` 下一级和二级目录
   - 对每个主要目录，读取入口文件（index.ts/index.js）获取导出内容
   - 识别目录的真实职责（不仅是文件名推断）

3. **识别代码模式**
   - 检测状态管理方案及其使用模式（如 valtio proxy、zustand slice）
   - 检测网络请求封装层（找到统一的 request/fetch wrapper）
   - 检测路由方案和页面注册方式
   - 检测组件库使用模式
   - 检测工具函数和 hooks 的组织方式

4. **生成精准内容**
   按以下优先级逐一生成：

   a. **glossary.md** — 术语映射
      - 从目录名、README、注释中提取业务术语
      - 建立"产品词 → 代码路径"的映射关系
      - 标注模块间的依赖方向

   b. **code-standards.md** — 代码规范增强
      - 基于实际代码风格推断编码约定
      - 从 ESLint/Prettier 配置提取硬性规则
      - 识别项目特有的编码模式并文档化

   c. **agent-capabilities.md** — 能力约束增强
      - 找到实际的请求封装并标注使用方式
      - 找到状态管理的 store 创建模式
      - 找到路由配置方式和页面注册方式

   d. **module-map** — 为核心模块生成详细地图
      - 只为体量大于 10 个文件的模块生成独立 module-map
      - 标注核心文件、公共接口、依赖关系

### 阶段二：持续维护（日常对话中触发）

当在普通开发对话中发现以下情况时，**主动建议**（不自动修改）：

1. **新模块未映射**
   - 用户操作的代码路径不在 overview.md 或 glossary.md 中
   - 建议：「发现新模块 `src/xxx/`，是否需要我更新 glossary 并生成 module-map？」

2. **术语未收录**
   - 用户使用的业务词在 glossary 中找不到映射
   - 建议：「术语"xxx"在 glossary 中未收录，是否要添加映射？」

3. **规则可能过期**
   - 检测到代码模式与已有规则描述不一致（如规则写"使用 Redux"但代码已迁移到 Zustand）
   - 建议：「检测到 xxx 规则可能已过期，是否需要更新？」

## 输出格式规范

### glossary 条目格式

```markdown
### 术语名称

- **别名**: 用户可能使用的其他叫法
- **主要目录**: `src/具体路径/`
- **说明**: 一句话描述该模块的核心职责
- **依赖**: 依赖的其他模块
- **被依赖**: 被哪些模块使用
```

### module-map 格式

```markdown
# 模块名 Module Map

## 概述
一段话描述模块的业务定位和技术方案。

## 目录结构
（实际扫描的树形结构）

## 核心文件
| 文件 | 职责 | 复杂度 |
|------|------|--------|

## 公共接口
（该模块向外暴露的 export）

## 数据流
（描述数据在模块内的流转方式）

## 依赖关系
- 上游：
- 下游：
```

### 规则增强格式

```markdown
## 具体规则名称

### 强制要求（MUST）
- 基于实际代码模式推断的硬性规则

### 推荐实践（SHOULD）
- 从优秀代码片段中提炼的最佳实践

### 禁止事项（MUST NOT）
- 从代码审查历史或 linter 配置中提取的禁止项

### 示例
（从实际项目代码中提取的正反例）
```

## 约束

1. **只建议不强写** — 阶段二中发现的更新需求，必须征得用户同意后才写入
2. **不改业务代码** — 本 Skill 只操作 `.agents/` 目录下的文件
3. **渐进式生成** — 不要一次性生成所有内容，优先生成 glossary → 再 rules → 再 module-map
4. **可验证性** — 生成的每条规则/映射都必须能在代码中找到对应证据
5. **保留人工标注** — 如果用户手动编辑过某个文件，合并时保留用户内容
