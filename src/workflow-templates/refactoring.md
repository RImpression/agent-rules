# Refactoring Workflow

## 目标

提供安全的代码重构流程，确保每次重构都：有测试保障 → 分步进行 → 逐步验证，避免"大刀阔斧改完一堆报错"。

## 适用场景

- 代码可读性差，需要改善命名/结构
- 模块职责不清，需要拆分或合并
- 重复代码过多，需要抽取公共逻辑
- 技术债务清理（如：废弃 API 替换、依赖升级适配）
- 性能优化（涉及代码结构调整）

## 工作流步骤

### Step 1: 现状评估

**目标**：明确重构范围和当前测试覆盖情况。

**执行动作**：
1. 确定重构目标（改善什么问题）
2. 定位涉及的文件和模块
3. 检查现有测试覆盖率：
   - 哪些文件/函数有测试
   - 哪些没有测试（重构前需要补充）
4. 识别该代码的上下游依赖（谁调用它、它调用谁）

**产出**：
- 重构目标（一句话）
- 涉及文件清单
- 测试覆盖现状
- 依赖关系图

**完成标准**：清楚"改哪里"和"现在有没有测试兜底"。

---

### Step 2: 补充测试保障

**目标**：确保重构前有足够的测试覆盖，作为安全网。

**执行动作**：
1. 为**未覆盖但即将重构**的代码补充测试
2. 测试应验证当前行为（而非期望行为），确保重构后行为不变
3. 运行补充的测试，确认全部通过

**跳过条件**：如果现有测试已完全覆盖重构范围，可跳过此步。

**完成标准**：重构涉及的核心逻辑都有测试覆盖。

---

### Step 3: 制定重构计划

**目标**：将重构拆解为可独立验证的小步骤。

**执行动作**：
1. 将大重构拆分为若干小步骤，每步：
   - 改动最小化
   - 独立可验证（改完能跑通测试）
   - 不混合功能变更
2. 确定执行顺序（通常从底层/被依赖方开始）
3. 标注每步的风险点

**产出**：
```markdown
## 重构计划

**目标**：<描述>

### 步骤拆解
1. <步骤1描述> — 影响文件: ...
2. <步骤2描述> — 影响文件: ...
3. <步骤3描述> — 影响文件: ...

### 执行顺序理由
...
```

**门控条件**：计划需用户确认后执行。

---

### Step 4: 分步执行

**目标**：按计划逐步重构，每步后验证。

**执行动作**：
对计划中的每一步：
1. 执行该步的代码改动
2. 运行 typecheck
3. 运行相关测试
4. 确认通过后再进入下一步

**约束**：
- 严格按计划执行，不"顺手"做计划外的改动
- 每步改动后必须验证，不积攒多步一起验证
- 如果某步导致测试失败，先修复再继续
- 不在重构中混入功能变更（如新增功能、修 bug）

---

### Step 5: 完整验证

**目标**：所有重构步骤完成后，做全面验证。

**执行动作**：
1. 运行完整测试套件
2. typecheck 全量检查
3. lint 检查
4. 对比重构前后的公共接口，确认未改变对外行为：
   - 导出的函数签名是否一致
   - 类型定义是否兼容
   - 调用方是否需要适配

**门控条件（必须全部通过）**：
- [ ] 全量测试通过
- [ ] typecheck 无错误
- [ ] lint 无新增警告
- [ ] 公共接口未发生 breaking change（除非用户明确要求）

---

### Step 6: 总结

**目标**：输出重构摘要和后续建议。

**执行动作**：
1. 汇报重构结果
2. 如果发现了新的技术债务，记录供后续处理
3. 建议是否需要更新 `.agents/` 规范（如模块结构变了）

**产出**：
```markdown
## 重构摘要

**目标**：<重构目标>
**范围**：<涉及模块>
**步骤数**：<n 步>
**改动统计**：<新增/修改/删除文件数>
**验证结果**：全部通过
**后续建议**：<如有技术债务或规范需更新>
```

## 流程图

```
用户要求重构
     │
     ▼
┌───────────────────┐
│ Step 1: 现状评估    │ ← 确定范围 + 测试覆盖
└──────┬────────────┘
       │
       ▼
┌───────────────────┐
│ Step 2: 补充测试    │ ← 为安全重构建立保障网
└──────┬────────────┘     （已有覆盖则跳过）
       │
       ▼
┌───────────────────┐
│ Step 3: 制定计划    │ ← 拆小步、用户确认（gate）
└──────┬────────────┘
       │ 用户确认
       ▼
┌───────────────────┐
│ Step 4: 分步执行    │ ← 每步改动 + 验证
└──────┬────────────┘
       │
       ▼
┌───────────────────┐
│ Step 5: 完整验证    │ ← 全量测试（gate）
└──────┬────────────┘
       │
       ▼
┌───────────────────┐
│ Step 6: 总结        │ ← 摘要 + 后续建议
└───────────────────┘
```

## 约束

1. **测试先行** — 没有测试保障的代码，先补测试再重构
2. **小步快跑** — 每步改动最小化，逐步验证
3. **不混功能** — 重构中不引入新功能或修复 bug
4. **接口兼容** — 除非用户明确要求，不改变公共接口
5. **计划确认** — 重构计划需用户确认后才能执行
6. **循环保护** — 单步修复循环不超过 3 次
