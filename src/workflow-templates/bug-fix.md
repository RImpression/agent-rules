# Bug Fix Workflow

## 目标

提供结构化的 Bug 修复流程，确保每次修复都经过：复现 → 定位 → 修复 → 验证 的完整闭环，避免"改了一处坏了另一处"。

## 适用场景

- 用户报告了一个具体的 bug 或异常行为
- CI/CD 中出现了失败的测试用例
- 线上监控告警需要排查修复
- 代码审查中发现的逻辑错误

## 工作流步骤

### Step 1: 复现与确认

**目标**：确认 bug 确实存在，明确复现路径。

**执行动作**：
1. 理解用户描述的问题现象（错误信息、异常行为、期望行为）
2. 确定复现条件：
   - 输入数据/参数
   - 操作路径/步骤
   - 环境条件（浏览器、Node 版本等，如果相关）
3. 尝试通过阅读代码或运行测试来确认问题
4. 如果无法确认，向用户询问补充信息

**产出**：
- 问题描述（一句话总结）
- 复现步骤
- 实际行为 vs 期望行为

**完成标准**：能够明确指出"在什么条件下会出现什么错误"。

---

### Step 2: 定位根因

**目标**：找到引发 bug 的根本原因代码，而非表面症状。

**执行动作**：
1. 根据复现路径，利用 `overview.md` 和 `glossary.md` 定位相关模块
2. 追踪代码执行链路：入口 → 中间处理 → 出错点
3. 区分根因与症状：
   - 根因：导致问题的**真正原因**（如：数据未做空值检查）
   - 症状：用户看到的**表面现象**（如：页面白屏）
4. 检查是否有相关的已有测试覆盖该路径

**产出**：
- 根因定位：具体文件、行号、错误逻辑描述
- 影响范围评估：该代码被哪些其他模块依赖

**完成标准**：能够解释"为什么这段代码会导致那个现象"。

---

### Step 3: 方案设计

**目标**：确定修复方案，评估风险。

**执行动作**：
1. 提出修复方案（可能有多个方案）
2. 评估每个方案的：
   - 改动范围（改几个文件、几行代码）
   - 风险点（是否可能影响其他功能）
   - 兼容性（是否需要考虑向后兼容）
3. 如果改动范围较大或有风险，**暂停并向用户确认方案**

**产出**：
- 推荐方案 + 理由
- 需要修改的文件清单
- 潜在风险说明

**门控条件**：
- 如果改动超过 3 个文件，或涉及公共模块/接口变更 → 必须等用户确认
- 如果改动仅限 1-2 个文件且风险低 → 可直接进入实现

---

### Step 4: 实现修复

**目标**：按方案修复代码。

**执行动作**：
1. 按方案修改代码
2. 遵循项目 `code-standards.md` 中的编码规范
3. 确保修改的最小化原则——只改必要的部分
4. 如果修复逻辑较复杂，添加注释说明"为什么这样改"

**约束**：
- 不做超出修复范围的重构
- 不顺手修"看起来不好但和 bug 无关"的代码
- 如果发现需要额外重构才能修复，回到 Step 3 重新评估

---

### Step 5: 验证

**目标**：确认 bug 已修复且未引入新问题。

**执行动作**：
1. **类型检查** — 运行 typecheck（如 `tsc --noEmit`），确保无类型错误
2. **Lint 检查** — 确认无新增 lint 警告/错误
3. **相关测试** — 运行与修改文件相关的测试用例
4. **回归验证** — 确认修复没有破坏原有功能：
   - 检查调用该函数/组件的其他地方是否仍正常
   - 如果有全局测试，运行完整测试套件
5. **补充测试**（如果原有测试未覆盖此场景）— 为 bug 场景添加测试用例，防止回归

**门控条件（必须全部通过）**：
- [ ] typecheck 无错误
- [ ] lint 无新增警告
- [ ] 相关测试全部通过
- [ ] bug 的复现路径不再复现

**如果验证失败**：
- 回到 Step 4 修正代码
- 最多循环 3 次，超过则暂停并向用户汇报困难点

---

### Step 6: 总结

**目标**：清晰汇报修复结果。

**执行动作**：
1. 输出修复摘要：
   - 根因是什么
   - 改了什么
   - 怎么验证的
2. 如果发现了潜在的同类问题（如：其他地方也缺少空值检查），建议用户关注
3. 如果本次修复暴露了规范缺失（如：应该有"必须做空值检查"的规则），建议更新 `.agents/rules/`

**产出**：
```markdown
## Bug Fix 摘要

**问题**：<一句话描述>
**根因**：<根因定位>
**修复**：<改动说明>
**验证**：<通过了哪些检查>
**建议**：<后续可关注的事项（可选）>
```

## 流程图

```
用户报告 bug
     │
     ▼
┌─────────────┐
│ Step 1: 复现 │ ← 无法复现则向用户追问
└──────┬──────┘
       │
       ▼
┌──────────────┐
│ Step 2: 定位 │ ← 利用 overview/glossary 做模块映射
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Step 3: 方案 │ ← 高风险改动需用户确认（gate）
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Step 4: 实现 │ ← 最小化改动，遵循 code-standards
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Step 5: 验证 │ ← 必须通过所有检查（gate）
└──────┬───────┘     失败 → 回到 Step 4（最多 3 次）
       │
       ▼
┌──────────────┐
│ Step 6: 总结 │ ← 输出修复摘要 + 后续建议
└──────────────┘
```

## 约束

1. **最小改动原则** — 只修复 bug 本身，不做无关重构
2. **必须验证** — 不允许跳过 Step 5 的验证环节
3. **循环保护** — Step 4 ↔ Step 5 循环不超过 3 次，超过则暂停求助
4. **透明汇报** — 每一步的结论对用户可见，不"静默"跳过步骤
5. **规范联动** — 修复过程中遵循 `code-standards.md`，发现规范缺失时建议更新
