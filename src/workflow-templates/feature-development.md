# Feature Development Workflow

## 目标

提供结构化的新功能开发流程，确保每次开发都经过：分析 → 设计 → 实现 → 验证 → 收尾 的完整闭环，避免"直接上手写、写完发现方向错"。

## 适用场景

- 用户提出一个新功能需求
- 需要新增页面、组件、API 等
- 改动涉及多个模块的协调
- 需求描述较为复杂或模糊

## 工作流步骤

### Step 1: 需求分析

**目标**：确认需求范围，识别影响的模块。

**执行动作**：
1. 提取需求中的核心功能点
2. 利用 `overview.md` 和 `glossary.md` 定位相关模块
3. 评估影响范围：需要新增/修改哪些文件
4. 识别依赖：是否需要新增依赖包、新建接口等
5. 如果需求模糊，列出待确认的问题向用户询问

**产出**：
- 功能点清单
- 涉及模块/文件列表
- 待确认问题（如有）

**完成标准**：能够清晰回答"要做什么"和"在哪里做"。

---

### Step 2: 方案设计

**目标**：输出具体实现方案，获得用户确认。

**执行动作**：
1. 设计整体技术方案：
   - 数据结构/类型定义
   - 模块间的调用关系
   - 状态管理方式（如涉及）
   - 接口设计（如涉及后端交互）
2. 列出具体的文件变更计划：
   - 新增文件（路径 + 职责描述）
   - 修改文件（具体改动点）
3. 标注风险点和设计取舍

**产出**：
```markdown
## 实现方案

### 新增文件
- `src/xxx/yyy.ts` — 描述

### 修改文件
- `src/xxx/zzz.ts` — 改动点

### 技术方案
- 数据流：...
- 状态管理：...

### 风险点
- ...
```

**门控条件**：必须等用户确认方案后才能进入实现。

---

### Step 3: 逐步实现

**目标**：按方案逐一实现各个文件。

**执行动作**：
1. 按照方案中的文件清单逐个实现
2. 实现顺序建议：
   - 类型/接口定义 → 工具函数 → 核心逻辑 → UI 组件 → 串联/路由
3. 每完成一个核心文件，运行 typecheck 确认无类型错误
4. 遵循项目 `code-standards.md` 中的编码规范
5. 添加必要的导入语句和依赖

**约束**：
- 不偏离已确认的方案，如需调整先告知用户
- 保持每次改动的原子性，避免一次改太多文件
- 新增的公共接口需要导出并在适当位置注册

---

### Step 4: 验证

**目标**：确认功能实现正确且不破坏已有功能。

**执行动作**：
1. **类型检查** — 运行 typecheck，确保无类型错误
2. **Lint 检查** — 确认无新增 lint 警告/错误
3. **功能测试** — 为新功能编写测试用例（单元测试或集成测试）
4. **回归测试** — 运行已有测试套件，确认未破坏原有功能
5. **手动验证**（如适用）— 描述如何手动验证功能是否正常

**门控条件（必须全部通过）**：
- [ ] typecheck 无错误
- [ ] lint 无新增警告
- [ ] 新增测试通过
- [ ] 已有测试未受影响

**如果验证失败**：
- 回到 Step 3 修正代码
- 最多循环 3 次，超过则暂停并向用户汇报

---

### Step 5: 收尾

**目标**：整理变更，更新相关文档。

**执行动作**：
1. 检查是否需要更新 `.agents/` 相关文件：
   - 新模块是否需要添加到 `glossary.md`
   - 新模式是否需要补充到 `code-standards.md`
2. 输出功能开发摘要

**产出**：
```markdown
## Feature 开发摘要

**功能**：<功能名称>
**范围**：<涉及模块>
**新增文件**：<列表>
**修改文件**：<列表>
**测试覆盖**：<测试文件路径>
**后续建议**：<如有>
```

## 流程图

```
用户提出需求
     │
     ▼
┌───────────────┐
│ Step 1: 分析   │ ← 需求模糊则追问
└──────┬────────┘
       │
       ▼
┌───────────────┐
│ Step 2: 设计   │ ← 必须用户确认（gate）
└──────┬────────┘
       │ 用户确认
       ▼
┌───────────────┐
│ Step 3: 实现   │ ← 逐文件实现，每步 typecheck
└──────┬────────┘
       │
       ▼
┌───────────────┐
│ Step 4: 验证   │ ← 必须通过所有检查（gate）
└──────┬────────┘     失败 → 回到 Step 3
       │
       ▼
┌───────────────┐
│ Step 5: 收尾   │ ← 摘要 + 规范更新建议
└───────────────┘
```

## 约束

1. **方案先行** — 禁止跳过 Step 2 直接写代码
2. **用户确认** — 方案必须经用户确认后才能实现
3. **逐步实现** — 不要一口气生成所有文件，按优先级逐步推进
4. **必须验证** — 不允许跳过 Step 4 的验证环节
5. **循环保护** — Step 3 ↔ Step 4 循环不超过 3 次
6. **规范遵循** — 全程遵循 `code-standards.md` 和 `agent-capabilities.md`
